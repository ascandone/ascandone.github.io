Why React?

If you have dealt with html using vanilla javascript you have probably dealt with the following problems:

  • No code reuse at all!
  • Complexity of the code grows exponentially as the app grows
  • Can't reliably test the code
  • Application logic and presentation not always separated (state often resides IN the DOM)

You might have used libraries like jquery, but it merely provides shortcuts for the common use cases, without actually solving those problems. The approach is exactly the same: imperatively manipulate the DOM.

React (among others) offers a radically different approach. That requires a small shift in your mindset, [TODO: fun, clean, elegant]

The virtual DOM

The concept behind the hood is quite simple. Let's consider the following html node:

<div class="header">
  <h2>Hello</h2>
  <p>World!</p>
</div>

You can model it in React as:

// I know it looks gross - for now!
const App = React.createElement('div', { className: 'header' }, [
  React.createElement('h2', null, 'Hello'),
  React.createElement('p', null, 'World'),
])

It simply instanciates (roughly) something like:

// A plain old tree data structure
{
  type: 'div',
  props: {
    className: 'header',
    children: [
      { type: 'h2', props: { children: ['Hello'] } },
      { type: 'p', props: { children: ['World!'] } },
    ],
  },
}

But of course we haven't touched the the DOM yet. Here's the magic:

render(App, document.getElementById('root'))

actually renders the tree we built inside the <div id="root"></div> node. But most importantly (and that's the point of the whole thing): subsequent renders calculate the miniminal changes on the DOM [TODO: complete] Note that DOM manipulation is usually the bottleneck in web applications perfomance.

This is, however, particularly verbose and unclear, so it is usually used an extension of javascript called jsx, that offers an handy syntactic sugar to write the previous code in the following way:

// the parenthesis is just needeed to prevent js inserting the ";" at the end of the line
// "class" attribute is called "className" to prevent grammar ambiguity (class is a reversed js keyword)
const App = (
  <div className="header">
    <h2>Hello</h2>
    <p>World!</p>
  </div>
)

Mmh, that's not good. Here we are repeating three times some very similar code. Copy-pasting the same code multiple times doesn't look that good to me. What if I want to change, or even just refactor it? I would have to change each one of those columns, and do the same thing over and over, even just to introduce a smal change. And what if I used that component in many different places?

It would take a very small time to mess up everything. And what's the simplest computer-sciency pattern to solve those problems?

Functions, of course!

Since we modeled an html node just as a javascript value, we also return it from a function. One way of designing it could be this:

const FeatureGrid = ({ header, content }) => (
  <div className="c-feature-grid__item">
    <h3 className="display-as-paragraph">
      <strong>{header}</strong>
    </h3>
    <p className="c-feature-grid__item__copy">{content}</p>
  </div>
)

The square brackets inside the html part are the jsx equivalent of string interpolation: inside the brackets it's just javascript. [TODO: explain better; use for props, need to be element or arr element]

Now we could of course use our component calling the function:

// Spoiler: don't actually do this
FeatureGrid({
  header: 'Conversations, organized',
  content:
    'Instead of a single overstuffed inbox, conversations in Slack happen in dedicated spaces called channels.',
})

But since it would be hardly readable to mix this syntax and jsx syntax, we actually call it this way, as it was regular html:

<div className="c-feature-grid v--three v--imageicon u-text--left">
  <FeatureGrid
    header="Conversations, organized"
    content="Instead of a single ..."
  />
  <FeatureGrid header="..." content="..." />
  <FeatureGrid header="..." content="..." />
</div>

This code is actually not the same thing as the function call above, since it uses the React.createElement function. [TODO: explain better] You can think of it as a technical detail, but in case you are interested the motivation of this decision is explained here

What you do need to know, hovever, is that is mandatory that the Component is a pure function - a function in the mathematical sense of the term: given the same arguments, it returns the same values, without performing any side-effects, like mutating external data or event console.log-ing stuff.

Also, (function) components only take one argument - an object, with arbitrary fields (except the reserved key and ref) - called "props".

Don't worry, those are, in practice, not limitations, but best practices you would follow anyway for components

Note that the children will be passed to the components as the "children" prop, as an array of react elements. Hence we could have designed our component this way:

const FeatureGrid = ({ header, children }) => (
  <div className="c-feature-grid__item">
    <h3 className="display-as-paragraph">
      <strong>{header}</strong>
    </h3>
    <p className="c-feature-grid__item__copy">{children}</p>
  </div>
)
<FeatureGrid header="Conversations, organized">
  Instead of a single ...
</FeatureGrid>
